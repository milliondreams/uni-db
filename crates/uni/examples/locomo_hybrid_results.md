# LoCoMo hybrid retrieval — `similar_to` vs `uni.search`

Generated by `cargo run --release --example locomo_hybrid --features provider-onnx`
(`crates/uni/examples/locomo_hybrid.rs`).

- **Model**: real `aapot/bge-m3-onnx` (`EmbedHybrid`), CPU ONNX, HF hub cache.
- **Corpus**: first **3 LoCoMo conversations** pooled = **1,451 dialogue turns**,
  each ingested as one `Doc` carrying **three retrieval signals**:
  - `emb` — BGE-M3 dense vector, HNSW index (m=16, efc=100, cosine)
  - `text` — raw turn text, BM25 **FULLTEXT** index
  - `sparse` — BGE-M3 learned-sparse (SPLADE) weights, sparse inverted index
- **Queries**: first **5 LoCoMo questions**, **top-5** per query.

This is the complement to `locomo_profile.rs`: that harness profiles each head in
its *own* single-index DB; this one puts all three arms on **one node** so the two
hybrid idioms can be compared on identical data.

## The two idioms exercised

| Idiom | Cypher | Semantics |
|---|---|---|
| Per-row scalar | `similar_to(d.emb, $q)` in `RETURN ... ORDER BY` | Scores every already-bound node (full label scan). Score = raw metric-aware cosine in `[0,1]`. |
| Top-K procedure | `CALL uni.search('Doc', {vector, fts[, sparse]}, $qtext, $qvec, k, null, {method:'rrf'[, sparse_query:$sq]})` | Runs each arm's index, fuses ranked lists with RRF. |

**Score scales are not comparable across idioms.** `similar_to` returns cosine
(~0.6–0.76 here). RRF returns `Σ 1/(60+rank)` per doc across arms (~0.016 per arm,
so ~0.03 two-arm / ~0.05 three-arm at rank 1) — small by construction, rank-based,
and robust to the fact that cosine `[0,1]` and BM25 (unbounded) live on different
scales. The `vec` / `fts` / `sparse` columns show each arm's own score; a blank
column means that doc did **not** land in that arm's over-fetch window (RRF still
fuses whatever ranks it did earn).

## Headline finding — hybrid rescues literal-keyword hits

**Q1: "When did Caroline go to the LGBTQ support group?"**

| Rank | dense-only `similar_to` | hybrid 2-way `uni.search` |
|---|---|---|
| 1 | `t282` Melanie: "Thanks, Caroline! I really appreciate…" (0.762) | **`t2` Caroline: "I went to a LGBTQ support group yesterday…"** (vec=0.730, **fts=1.000**) |
| 2 | `t2` Caroline: "I went to a LGBTQ support group…" (0.731) | `t6` "The support group has made me feel accepted…" |

Dense-only ranks a generic thank-you turn (`t282`) **above** the gold answer (`t2`)
— the sentence embedding is dominated by conversational tone. The BM25 arm scores
`t2` at **1.0** on the literal phrase "LGBTQ support group", and RRF lifts it to
**#1**. This is the canonical case for hybrid over pure dense.

**Q4: "What did Caroline research?"** — dense-only never surfaces the gold turn in
top-5; the 2-way hybrid pulls in `t25` Caroline: "Researching adoption agencies —
it's been a dream…" (fts=0.816) via the lexical arm.

## What the sparse (3-way) arm adds

Adding `{sparse: 'sparse'}` + `options.sparse_query` populates `sparse_score`
(BGE-M3 SPLADE dot product) and re-fuses. It mostly **reinforces** the 2-way top
hits (Q1/Q2/Q4 keep the same #1) while reshuffling the tail: e.g. Q1 promotes
`t193` above `t29`, and Q2 surfaces `t276`/`t213` that neither dense nor FTS ranked
highly. On these short, high-overlap turns the learned-sparse and BM25 arms often
agree, so the marginal effect is smaller than the dense→+FTS jump — sparse pays off
more on longer / vocabulary-mismatched corpora.

## Query timings (497 questions × top-5 over 1,451 turns)

Single-threaded, sequential, CPU; **query-embedding cost excluded** (each mode is
timed over the *same* pre-embedded questions, so this isolates retrieval).

| mode | mean (ms) | p50 (ms) | p95 (ms) | p99 (ms) | QPS |
|---|---:|---:|---:|---:|---:|
| dense-only `similar_to` | 20.85 | 20.76 | 22.38 | 24.47 | 48 |
| hybrid 2-way `uni.search` {vector, fts} | 7.67 | 7.50 | 9.19 | 9.91 | 130 |
| hybrid 3-way `uni.search` {vector, fts, sparse} | 11.18 | 10.93 | 13.68 | 14.53 | 89 |

**The per-row idiom is the slowest — by design.** `similar_to(d.emb, $q)` after
`MATCH (d:Doc)` is an O(N) **full label scan**: it cosine-scores all 1,451 turns
every query, so latency grows with corpus size and the distribution is very tight
(p99/p50 ≈ 1.18 — it does identical work each time). The two `uni.search` variants
hit the **HNSW ANN graph** (dense) and **inverted indexes** (BM25 / sparse), which
are sublinear, so 2-way is **~2.7× faster** than dense-only despite fusing *two*
arms plus loading `node.title`/`node.text`.

Adding the learned-sparse arm (2-way → 3-way) costs **+3.5 ms mean** (7.67 → 11.18)
and drops throughput 130 → 89 QPS — the price of the extra SPLADE dot-product arm +
three-way RRF fusion.

> Takeaway: use `similar_to` to **score an already-small, graph-narrowed candidate
> set** (post-`MATCH`/traversal), not to scan a whole label. To *retrieve* top-K
> from a full label, `uni.search` (or `uni.vector.query`) is both faster and the
> intended tool. (Cross-check: `locomo_profile.rs` clocks the isolated dense-HNSW
> arm at ~4.7 ms and the sparse arm at ~5 ms in their own single-index DBs; the
> higher numbers here include RRF fusion and full `node` property materialization.)

The table above is from the **dense+FTS+sparse** configuration (no `tokens`
column). Adding the multivector head changes these numbers — see below.

## Adding the multivector (ColBERT) head

`similar_to` has **no** multivector mode (its four modes are vec-vec, vec-string
auto-embed, string-string FTS, and multi-source RRF fusion). Late interaction is
reachable only two ways, both added here on the same `Doc`:

1. **Standalone MUVERA/FDE** — `tokens` typed `List(Vector(1024))` with a
   `VectorAlgo::Muvera{k_sim:4,reps:8,d_proj:8,inner:Flat}` index, queried via
   `CALL uni.vector.query('Doc','tokens',$mvq,k)` (FDE retrieve + MaxSim rescore).
2. **MaxSim reranker on hybrid** — `CALL uni.search('Doc',{vector:'emb',fts:'text'},
   $qtext,$qvec,k,null,{reranker:'maxsim', reranker_property:'tokens',
   maxsim_query:$mvq})`. `reranker_property` (not the `vector:` key) names the
   doc-side token vectors; with a reranker active, `score` == MaxSim `rerank_score`.

### 5-mode timings (497 Q × top-5, 1,451 turns, WITH the `tokens` column)

**Leak-free (`YIELD vid` — retrieval only, no node/property materialization).**
This is the correct table; see the note below on why the first cut was inflated.

| mode | mean (ms) | p50 | p95 | p99 | QPS |
|---|---:|---:|---:|---:|---:|
| hybrid 2-way {vector, fts} | **15.47** | 15.32 | 17.98 | 19.26 | **65** |
| dense-only `similar_to` | 17.36 | 17.12 | 19.43 | 22.41 | 58 |
| hybrid 3-way {vector, fts, sparse} | 26.49 | 26.27 | 30.97 | 32.58 | 38 |
| **hybrid + MaxSim rerank** | 53.40 | 53.07 | 65.74 | 70.33 | **19** |
| **multivec (MUVERA + MaxSim)** | 53.89 | 53.43 | 72.98 | 79.20 | **19** |

Clean picture: **2-way hybrid is the fastest** (65 QPS) — HNSW ANN + BM25 inverted
beat the O(N) full-label scan of `similar_to` (58 QPS). The **sparse arm adds ~11 ms**
(2-way → 3-way). The two **ColBERT late-interaction modes are the tier apart** (~54 ms,
19 QPS, ~3.5× the 2-way): standalone MUVERA and MaxSim-rerank cost essentially the
same here — MaxSim scoring dominates either way.

**Why the first table was wrong (and how much).** My initial timings used the
display query (`YIELD node → RETURN node.title, node.text`), which materializes the
full node including the ~145 KB/turn `tokens` list — the projection leak. Re-timing
with `YIELD vid` removes it. Measured within the *same* `tokens`-carrying DB:

| mode | YIELD node (leaked) | YIELD vid (clean) | leak tax |
|---|---:|---:|---:|
| dense-only `similar_to` | 18.35 | 17.36 | ~0 (never reads `tokens`) |
| hybrid 2-way | 22.17 | 15.47 | **−30 %** |
| hybrid 3-way | 32.54 | 26.49 | −19 % |
| multivec | 54.97 | 53.89 | ~0 |
| hybrid + MaxSim | 50.78 | 53.40 | ~0 (noise) |

The leak tax lands almost entirely on the **cheap RRF hybrid modes**, where node
materialization is a large fraction of a ~15–30 ms query. For the **late-interaction
modes** the MaxSim/MUVERA compute (~54 ms) dwarfs property loading, so the leak is
negligible there. `dense-only` is unaffected because it only returns `title`/`text`,
never `tokens`.

> **Variance caveat:** these are single-threaded CPU numbers taken while a co-tenant
> job contended for RAM/CPU; `dense-only` alone read 20.85 / 18.35 / 17.36 ms across
> three runs (~±15 %). Treat absolutes as approximate; the *within-run* deltas
> (leak tax, mode-to-mode ratios) are the reliable signal.

### Quality (Q1 "When did Caroline go to the LGBTQ support group?")

- **Standalone MUVERA** ranked `t282` (generic thank-you) #1 and dropped the gold
  turn `t2` out of the top-5 — the FDE is a *coarse approximation* of MaxSim.
- **Hybrid + MaxSim rerank** put the gold `t2` at #1 (MaxSim=9.45). Late
  interaction shines as a **precision reranker over a good candidate set**, not as
  a standalone approximate retriever.

### Anomaly investigation — isolated with a minimal repro

Co-locating the `tokens` column + MUVERA index perturbed the *other* arms vs the
no-`tokens` run. Two effects, both now explained (minimal repro:
`examples/hybrid_multiindex_repro.rs`, synthetic 4-d vectors, single variable =
the MUVERA index):

1. **Latency (real):** 2-way 7.67 → 22.17 ms, 3-way 11.18 → 32.54 ms (~3×).
   `uni.search`'s `YIELD node → RETURN node.title, node.text` materializes the full
   node, now including the ~145 KB/turn `tokens` list — the known projection-leak
   surface. `YIELD vid` (+ a follow-up `MATCH` for display fields) avoids it.

2. **Dense `vector_score` scale — CONFIRMED normalization bug (HNSW-only):** under
   an **HNSW** `emb` index, `uni.search` returns the **raw cosine** as
   `vector_score` when the label has *only* that index, but flips to the
   metric-aware **`(1+cos)/2`** once a second vector index (MUVERA on `tokens`) is
   added. Under **Flat** it is always `(1+cos)/2`. The repro shows this on synthetic
   data: HNSW no-index `vector_score(d1)=0.0` (raw cos 0) vs with-MUVERA `=0.5`;
   Flat is `0.5` in both. This exactly reproduces LoCoMo: no-index `vector_score(t2)
   = 0.73` ≈ raw cosine, with-MUVERA `= 0.8657 = (1+0.7314)/2`. `similar_to` returns
   the raw cosine in all cases — so `similar_to` and `uni.search` also disagree by
   this same normalization, independent of the second index.

**Retraction:** the earlier note that the FTS arm "surfaced lexically-unrelated
turns" is **not** an index-corruption effect — the repro asserts the FTS-positive
set is invariant to the second index (Flat and HNSW). The remaining LoCoMo
fused-top-5 difference between runs is not fully isolated and is **not** attributed
to FTS corruption; the confirmed defect is the HNSW `vector_score` scale above.

**Impact:** ranking within the dense arm and pure-RRF fusion are preserved (both
normalizations are monotonic in cosine). The bug bites **weighted fusion**
(`method:'weighted'`, which blends absolute `vector_score`) and any **threshold**
on `vector_score`, where a scale that silently depends on an unrelated index is
wrong. Suggested fix: make the HNSW retrieval path apply the same metric-aware
score conversion as Flat/`uni.vector.query`, regardless of how many vector indexes
the label has.

## Notes / caveats

- **RRF point-context**: `similar_to([sources],[queries])` (the *expression* hybrid
  form) degenerates to equal-weight averaging per row and emits an `RrfPointContext`
  warning — use `uni.search` (as here) for genuine cross-arm rank fusion, or
  `similar_to(..., {method:'weighted'})` for explicit per-arm weights.
- Single-threaded CPU ONNX; embedding (~10 turns/s) dominates wall-clock, not retrieval.
- This is a **qualitative** demo (does hybrid change the ranking, and sensibly?),
  not a recall@k / nDCG benchmark against LoCoMo gold answers.
- `uni.search` `vector_score` / `fts_score` / `sparse_score` read back as `f64`;
  the demo binds `db.session()` to a local before building the query so the
  conditional `sparse_query` param does not outlive a temporary session (E0716).
