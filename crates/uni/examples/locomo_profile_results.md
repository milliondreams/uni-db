# LoCoMo BGE-M3 3-head runtime profile

Generated by `cargo run --release --example locomo_profile --features provider-onnx`
(`crates/uni/examples/locomo_profile.rs`).

- **Model**: real `aapot/bge-m3-onnx` (`EmbedHybrid`), CPU ONNX, HF hub cache.
- **Corpus**: first **3 LoCoMo conversations** pooled = **1,451 dialogue turns** indexed,
  **497 questions** issued as KNN queries, **k = 10**.
- **Method**: one shared hybrid embed pass produces all 3 heads; each head is then loaded
  as *precomputed* vectors into its own single-index DB (`embedding: None`) so per-head
  insert / build / footprint / query latency isolate index cost from the shared embed.

## Shared embedding (one hybrid pass emits all 3 heads)

| phase   | wall   | throughput      |
|---------|--------|-----------------|
| turns   | 135.2s | 10.7 turns/s    |
| queries | 10.1s  | 49.1 queries/s  |

Embedding cost is **not** separable per head — BGE-M3 computes dense, sparse and
multivector in the same forward pass.

## Per-head index + query performance

| head        | insert (ms) | index build (ms) | disk (MB) | vectors (MB) | mean (ms) | p50 (ms) | p95 (ms) | p99 (ms) | QPS | peak RSS (MB) |
|-------------|------------:|-----------------:|----------:|-------------:|----------:|---------:|---------:|---------:|----:|--------------:|
| dense       |         206 |              370 |      40.9 |          5.7 |      4.71 |     4.56 |     5.76 |     6.64 | 212 |         2,149 |
| sparse      |         178 |               65 |       3.4 |          0.2 |      5.05 |     4.88 |     6.48 |     7.25 | 198 |         2,170 |
| multivector |      28,777 |           20,553 |   1,133.3 |        211.5 |     94.71 |    93.65 |   126.03 |   140.55 |  11 |        13,508 |

- **dense** — HNSW (m=16, ef_construction=100, ef_search=100), Cosine.
- **sparse** — inverted index, f32, dim = 250,002 (XLM-RoBERTa vocab).
- **multivector** — MUVERA FDE (k_sim=4, reps=8, d_proj=8), Flat inner, Cosine.

## Findings

1. **Multivector is 1–2 orders of magnitude heavier on every axis**: ~140× insert, ~55×
   build, ~28× disk, ~37× vector bytes, ~19× query latency (95 ms vs ~5 ms), ~18× lower
   QPS, and ~6× peak process RSS (13.5 GB vs ~2.1 GB). Late-interaction fidelity is
   expensive.
2. **Dense ≈ sparse for query latency** (~5 ms, ~200 QPS) — both are single-vector ANN.
   Sparse has the **smallest footprint** (0.2 MB vectors ⇒ ~36 nonzeros/turn; BGE-M3
   sparse weights are highly peaked), dense is a fixed 1,451×1024 floats (5.7 MB).
3. **Index build**: sparse cheapest (65 ms, no ANN structure), dense HNSW 370 ms,
   multivector MUVERA 20.5 s.
4. **Embedding throughput (10.7 turns/s on CPU) is the shared ceiling** regardless of head.

## Operational note (surfaced during the run)

uni's default `commit_timeout` is **5 s**. Dense/sparse batches commit well under that,
but bulk multivector `List<Vector>` writes to Lance do not — a single large commit times
out and the cancelled commit panics inside `lance-datafusion`. This harness works around
it with batched commits (200 docs/tx) **and** `UniConfig.commit_timeout = 300s`. Bulk
multivector ingestion needs one or both in production.

## Caveats

- Latencies are **single-threaded, sequential** query timings on CPU ONNX; not concurrent.
- Corpus is **pooled** across 3 conversations into one index/head (bigger, more stable
  latency signal); it is not per-conversation retrieval.
- This is a **runtime** profile — it does not score retrieval quality (recall/nDCG).
- Peak RSS is process-level (includes the ~2 GB model); most meaningful as the relative
  multivector blow-up.
