# Migration: wheel-matrix collapse (12 → 6 wheels) — `uni-db 1.2.0`

**Status:** Active
**Date:** 2026-04-29
**Driven by:** `uni-xervo` 0.7.0 → 0.9.0 feature-surface collapse
**Supersedes:** `docs/proposals/python-wheel-matrix.md` (kept as historical record)

---

## TL;DR

`uni-db 1.2.0` ships **6 wheels** instead of 12. They map cleanly to two questions:

1. **Do I need local LLM inference (candle/mistralrs)?** — if yes, pick a default wheel; if no, pick a slim `-onnx` wheel and skip ~hundreds of MB of binary.
2. **What hardware accelerator?** — CPU (no suffix), NVIDIA (`-cuda`), Apple GPU/ANE (`-metal`).

| Wheel | Local providers | Remote APIs | Accelerator |
|---|---|---|---|
| `uni-db` *(default)* | candle + mistralrs + ONNX | all 8 | CPU |
| `uni-db-onnx` *(slim)* | ONNX only | all 8 | CPU |
| `uni-db-cuda` | candle + mistralrs + ONNX | all 8 | NVIDIA CUDA |
| `uni-db-metal` | candle + mistralrs + ONNX | all 8 | Apple GPU/ANE |
| `uni-db-onnx-cuda` *(slim+CUDA)* | ONNX only | all 8 | NVIDIA CUDA |
| `uni-db-onnx-metal` *(slim+Metal)* | ONNX only | all 8 | Apple GPU/ANE |

The Python API is identical across all six.

## Migration mapping (old → new wheel)

| Old wheel | New wheel | Notes |
|---|---|---|
| `uni-db` | `uni-db` | The default wheel now bundles all 11 providers (was: minimal — only `lance-backend` + `provider-gemini` + `provider-openai`). If you specifically need the old minimal build, install `uni-db-onnx`. |
| `uni-db-onnx` | `uni-db-onnx` | Now also includes all 8 remote API providers (previously ONNX-only). Slim local-inference variant. |
| `uni-db-onnx-cuda` | `uni-db-onnx-cuda` | Same shape, plus the 8 remote APIs. |
| `uni-db-onnx-metal` | `uni-db-onnx-metal` | Same shape, plus the 8 remote APIs. xervo 0.9.0's `gpu-metal` now activates `ort?/coreml`, so ORT actually reaches Apple GPU/ANE — previously it was silently CPU on Metal wheels. |
| `uni-db-fastembed` | `uni-db-onnx` | xervo 0.8.0 retired the FastEmbed wrapper and replaced it with a native ONNX `Embed` task on `local/onnx`. **All 25 FastEmbed alias strings still work** — just change `provider_id` from `"local/fastembed"` to `"local/onnx"`. The model itself is unchanged. |
| `uni-db-fastembed-cuda` | `uni-db-onnx-cuda` | Same migration as above; CUDA acceleration now goes through the ORT CUDA EP for embeddings (previously fastembed had its own CUDA path). |
| `uni-db-fastembed-metal` | `uni-db-onnx-metal` | Same migration; ORT CoreML EP handles Apple acceleration. |
| `uni-db-mistralrs` | `uni-db` | mistralrs is now part of the default wheel. |
| `uni-db-mistralrs-cuda` | `uni-db-cuda` | Combined with candle + ORT in one CUDA wheel. |
| `uni-db-mistralrs-metal` | `uni-db-metal` | Combined with candle + ORT (CoreML) in one Metal wheel. |
| `uni-db-all` | `uni-db` | The default wheel *is* "all providers, CPU" now. |
| `uni-db-all-cuda` | `uni-db-cuda` | Default wheel + CUDA. |
| `uni-db-all-metal` | `uni-db-metal` | Default wheel + Metal/CoreML. |

## Code changes for users still on `local/fastembed`

If your application catalogs an embedding alias like:

```python
# OLD (uni-db <= 1.1.x with fastembed wheel)
{"alias": "embed/default", "task": "Embed",
 "provider_id": "local/fastembed", "model_id": "BGESmallENV15", ...}
```

change `provider_id` to `"local/onnx"`:

```python
# NEW (uni-db >= 1.2.0)
{"alias": "embed/default", "task": "Embed",
 "provider_id": "local/onnx", "model_id": "BGESmallENV15", ...}
```

`model_id` (the alias enum string) is **unchanged** for all 25 FastEmbed presets. xervo 0.8.0 verified parity end-to-end against `fastembed-rs v5.13`: 22/25 models cosine = 1.000000, and the 3 dynamic-quantized variants land at ~0.99 (input-dependent quant scales).

The local cache directory moves from `.uni_cache/fastembed/<alias>/` to `.uni_cache/onnx-embed/<sanitized-repo>/`. Old caches can be deleted; the first call will re-download (~30s for BGE-small).

## What happened to the GPU EP features?

`uni-db 1.1.x` exposed nine `gpu-*` features mirroring uni-xervo's old surface (`gpu-tensorrt`, `gpu-coreml`, `gpu-rocm`, `gpu-directml`, `gpu-openvino`, `gpu-qnn`, `gpu-wgpu`, `gpu-cuda`, `gpu-metal`).

`uni-db 1.2.0` keeps two: `gpu-cuda` and `gpu-metal`. The other seven are still reachable for power users via:

```bash
# Use BYO ONNX Runtime at deploy time
cargo build --no-default-features \
  --features lance-backend,provider-onnx-dynamic,provider-...

# At runtime
export ORT_DYLIB_PATH=/path/to/your/libonnxruntime.so
```

This works for ROCm (AMD), OpenVINO (Intel), DirectML, QNN (Qualcomm), and any other ORT EP that ships through a vendor channel rather than pyke's prebuilt bundles. xervo 0.9.0 deleted the build-time MS-tarball fetcher (`build/ort_vendor.rs`) — vendor variability is now a runtime concern, not a build-time one.

## Picking a wheel

The probe inside the package will recommend a wheel for your host:

```python
from uni_db import recommend
print(recommend())
# -> "uni-db-cuda" on Linux/Windows with NVIDIA driver
# -> "uni-db-metal" on Apple Silicon
# -> "uni-db" elsewhere
```

Decision tree, by hand:

1. **Mac with M-series chip?** → `uni-db-metal`
2. **Linux/Windows with NVIDIA GPU?** → `uni-db-cuda`
3. **No GPU, want all 11 providers?** → `uni-db`
4. **No GPU, only need ONNX + remote APIs (smaller wheel)?** → `uni-db-onnx`
5. **Slim variant + GPU?** → `uni-db-onnx-cuda` or `uni-db-onnx-metal`

---

## Rationale (why collapse?)

The 12-wheel matrix was designed against uni-xervo 0.6.x: a separate FastEmbed provider, nine ONNX execution-provider features, and a build-time tarball fetcher for non-pyke EPs. uni-xervo 0.7.0 → 0.9.0 collapsed all of that into a single `local/onnx` provider with three tasks (raw, rerank, embed) and three orthogonal feature axes (provider × linking × acceleration). Carrying 12 wheels over a feature surface that no longer requires them was net cost, not net value.

See `uni-xervo/CHANGELOG.md` for the upstream story (0.7.0 unification, 0.8.0 fastembed retirement, 0.9.0 feature-surface collapse).
